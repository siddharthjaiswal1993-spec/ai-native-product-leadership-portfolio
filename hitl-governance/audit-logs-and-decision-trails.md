# Audit Logs and Decision Trails

**What to log, schema design, immutability, search, compliance requirements, and the PM's role in defining the audit specification**

---

## Why Audit Logs Are a First-Class Product Feature

In traditional software, audit logs are a compliance checkbox — something the security team asks for and engineering implements minimally. In AI-assisted enterprise products, the audit log is a core product feature.

When an AI system influences a consequential decision — a customer communication, a risk assessment, an identity provisioning action — the people accountable for that decision need to be able to reconstruct:
- What the AI recommended, and on what basis
- Who reviewed the recommendation
- What changes the reviewer made
- What action was taken
- What the outcome was

Without this trail, it is impossible to:
- Investigate a complaint from a customer about how a decision was made
- Defend an enterprise audit that asks how a provisioning configuration was authorised
- Train future models on the quality of historical decisions
- Identify systematic errors in AI recommendations (which requires knowing what was recommended, not just what was acted on)

The PM's role is to define what must be logged, to what granularity, and to ensure the audit log is accessible and usable — not just technically present.

## What to Log

### Minimum Required Events (Non-Negotiable)

| Event | Description |
|---|---|
| AI output generated | A finding, recommendation, draft, or configuration was generated by the AI system |
| AI output shown to reviewer | The output was surfaced to a human reviewer (in-app, email, API) |
| Reviewer action | The reviewer approved, edited, dismissed, or escalated the item |
| Content delta (if edited) | The original AI output and the reviewer-modified version |
| Action executed | The action approved by the reviewer was executed in a downstream system |
| Execution result | Success / failure / partial success of the executed action |
| Downstream system confirmation | Confirmation from the downstream system that the action was applied |

### Additional Events (Recommended)

| Event | Description |
|---|---|
| Retrieval query and results | What was retrieved from the knowledge base for this output (for RAG systems) |
| Confidence score | The AI system's confidence at the time of generation |
| Escalation events | To whom, when, and why an item was escalated |
| SLA events | SLA timer start, escalation threshold crossed, SLA expired |
| Rejection reason | Structured reason provided by the reviewer when dismissing or heavily editing |
| Model/prompt version | Which model and prompt version was used for this output |
| Data freshness at generation time | Timestamp of the most recent data refresh for key sources used in this output |

## Audit Log Schema

**Core event schema:**
```json
{
  "event_id": "uuid",
  "event_type": "ai_output_generated | reviewer_shown | reviewer_action | action_executed | escalation | sla_event",
  "timestamp": "ISO 8601 with timezone",
  "session_id": "uuid (groups all events for one AI-assisted workflow instance)",
  "entity_type": "account | deal | location | user | integration",
  "entity_id": "string",
  "actor_type": "ai_system | human",
  "actor_id": "string (system_id for AI; user_id for human — never raw PII in logs)",
  "action": "string (specific action taken)",
  "content_snapshot": {
    "original": "string (AI-generated content at time of event)",
    "modified": "string or null (reviewer-modified content, if applicable)"
  },
  "metadata": {
    "confidence_score": "number or null",
    "model_version": "string",
    "prompt_version": "string",
    "retrieval_sources": ["source_id"],
    "data_freshness": {"source_name": "ISO 8601 timestamp"}
  },
  "audit_context": {
    "rejection_reason": "string or null",
    "escalation_reason": "string or null",
    "approval_sla_status": "on_time | escalated | expired | null"
  },
  "environment": "production | staging | test",
  "trace_id": "string (distributed tracing ID for debugging)"
}
```

## Immutability Requirements

Audit logs must be immutable — once written, an audit log record cannot be modified or deleted by any user, including administrators. This is a security and compliance requirement.

**Immutability implementation options:**

**Append-only database:** Use an append-only data store (e.g., AWS QLDB, a Postgres table with no UPDATE/DELETE permissions, or an immutable audit log service). New records can be added; existing records cannot be changed.

**Cryptographic chaining:** Hash each log record and include the hash of the prior record in each new record (like a blockchain). Any modification to a prior record is detectable because the chain breaks. Used in high-compliance contexts.

**Write-once storage:** Write audit logs to write-once storage (e.g., S3 with Object Lock in WORM mode). Records in WORM storage cannot be overwritten or deleted before the retention period expires.

**Tamper detection:** Periodic integrity checks that re-hash stored records and compare against expected hash chains. Discrepancies trigger a security alert.

**Minimum:** Append-only database with role-based access control (no user has UPDATE or DELETE permission on the audit log table). Production (or WORM storage) for regulated industries.

## Search and Access

An audit log that cannot be searched is not useful. The PM must define the search requirements:

**Required search capabilities:**
- By entity (account_id, location_id, deal_id): "Show all AI-assisted decisions for Account X"
- By actor (user_id): "Show all decisions reviewed by CSM Y in the last 30 days"
- By event type: "Show all escalation events in the last 7 days"
- By time range: "Show all AI outputs generated between [START] and [END]"
- By outcome (rejected, escalated, auto-executed): "Show all items rejected in the last 30 days"

**Access control:**
- Individual users: can access audit logs for their own accounts and actions only
- Team leads / managers: can access audit logs for their team's accounts and actions
- Compliance / security: full audit log access with a documented access request process
- No user has DELETE access to the audit log

## Compliance Considerations

The specific compliance requirements for audit logs depend on the customer's industry and the type of data involved:

**SOC 2 Type II:**
- Requires audit logging of access to system components and data
- Log retention: typically 1 year minimum
- Access to logs must be controlled and logged

**GDPR / CCPA:**
- Audit logs that contain personal data (customer names, contact information) must be subject to data retention policies
- Right to erasure requests may require removing PII from audit logs while retaining the event record structure — plan for PII redaction capability in audit log schema design
- Log retention: comply with the minimum retention required for legitimate purposes; do not retain longer

**HIPAA (if customer is in healthcare):**
- Audit controls required for all PHI-containing system access
- Logs must be retained for 6 years minimum
- Access to logs must be restricted and logged

**FedRAMP (if customer is US government):**
- Specific audit log content requirements per NIST SP 800-53
- Logs must be protected against modification
- Specific retention requirements by log type

**PM's role:** Define the audit log schema with compliance requirements in mind. Work with the security and legal team to confirm the schema meets the requirements for your target customer industries. Do not leave audit log design to be "compliance-approved" as an afterthought — design it compliance-ready from the start.

## Retention Policy

| Log Type | Minimum Retention | Maximum Retention (before purge) |
|---|---|---|
| AI output generated | 24 months | 60 months |
| Reviewer actions (approvals, rejections) | 36 months | 84 months |
| Action execution records | 36 months | 84 months |
| Escalation events | 24 months | 60 months |
| Security-relevant events (unauthorized access attempts) | 84 months | Indefinite |

Retention periods start from the date of the event. Logs approaching the maximum retention age are archived before purge; the archive is retained for an additional 12 months before permanent deletion.

## The PM's Audit Spec Checklist

Before implementation, the PM should define and document:

- [ ] What events are logged (from the minimum required events list)
- [ ] What is included in each event's metadata
- [ ] Who has read access to the audit log, under what conditions
- [ ] How long logs are retained, by log type
- [ ] How PII in logs is handled (redaction strategy)
- [ ] How immutability is enforced
- [ ] How audit logs are searched (query interface requirements)
- [ ] What constitutes a compliance-relevant log event for the target customer industries
- [ ] How the audit log is included in the product's SOC 2 or equivalent certification scope
- [ ] How users can request an audit log export for their own account data (if required by data residency agreements)
